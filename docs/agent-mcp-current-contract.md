# Current Agent and MCP Contract

This document is the backend-owned current contract for Dirextalk Agent state, Native Agent, and external MCP access. It describes existing behavior; it does not add a compatibility surface.

## External MCP

- External MCP clients use the standard Streamable HTTP endpoint `POST /mcp`. `/_p2p/mcp` is unavailable.
- Requests are MCP JSON-RPC and currently support `initialize`, `tools/list`, and `tools/call`.
- Authentication is `Authorization: Bearer <agent_token>`. The owner access token is not accepted, bearer tokens in query parameters are rejected, and the inbound bearer token must not be forwarded in tool arguments or to downstream services.
- The endpoint validates `Origin`. `GET /mcp` returns method not allowed while server-to-client streaming is unused.
- Fixed `mcp.*` body actions are removed from `/_p2p/query` and `/_p2p/command`. Any `mcp.*` action identifiers that remain in backend packages are internal capability identifiers, not callable product actions.
- `POST /mcp` and Native Agent built-in Dirextalk tools share the backend `internal/dirextalkmcp` registry, schemas, pagination, room authorization, DTOs, and invocation service. Room IDs in `mcp_blocked_room_ids` are filtered from discovery and rejected on direct access.
- The only Cloud MCP capabilities are read-only `dirextalk_cloud_workloads_list`, `dirextalk_cloud_workloads_get`, and `dirextalk_cloud_status`. They return whitelisted Plan/Deployment/Service summaries, de-secretsed Job aggregate counts, and alert metadata; they never return Goal prompts, Plan narratives, outbox data, account metadata, credential/secret references, pairing data, or service secrets. No Cloud create, approval, networking, operation, or destroy tool is registered.

## Agent Room Status

- The configured `agent_room_id` is a real private Matrix room containing the owner and local `@agent:<server>` user.
- Bridge availability is Matrix room state type `io.dirextalk.agent.status`, state key `@agent:<server>`, with content field `online`.
- The running bridge publishes `online=true` or `online=false` through its Matrix session. Server startup or repair and `agent.config.update enabled=false` may publish `online=false` as a safe fallback.
- The server does not infer bridge availability from Agent configuration, `/sync`, realtime WebSocket lifetime, or Matrix presence. `sync.bootstrap` returns `agent_room_id`, not `agent_online`, and does not emit `agent.presence`.
- A non-YOLO local `dirextalk-connect` tool permission may use this private timeline only with an exact bridge-side `approval_owner_id`. The bridge emits ordinary `m.room.message` cards with msgtypes `io.dirextalk.agent.approval.request`, `io.dirextalk.agent.approval.response`, and `io.dirextalk.agent.approval.result` under the strict `io.dirextalk.agent_approval` envelope. A request exposes only an opaque approval UUID, kind `tool`, bounded tool name, and de-secreted summary; a response contains only that UUID plus `allow` or `deny`; a terminal result is `allowed`, `denied`, `expired`, or a bounded non-sensitive failure code. Connect binds the exact Matrix owner/room and resolves each request once. Message Server preserves these as Matrix timeline events only: it creates no ProductCore action, approval state, outbox, event projection, or authorization inference from card presence.

## Native Agent Split Boundary

- `dirextalk-message-server` continues to own the public `agent.*` ProductCore actions, owner authorization, `client.native_agent_stream` / `server.native_agent_stream.*` frames, Matrix Agent identity/room/status, and Dirextalk MCP. Chat and StreamChat execution can optionally be delegated to the independent `dirextalk-agent` gRPC service; model/config/Skill/Knowledge and the remaining runtime actions stay on the existing local Runner until their typed contracts are migrated.
- Remote Chat is disabled by default. Enabling it requires `P2P_AGENT_GRPC_ENABLED=true` plus `P2P_AGENT_GRPC_TARGET`, `P2P_AGENT_GRPC_CA_FILE`, `P2P_AGENT_GRPC_SERVER_NAME`, the canonical UUID `P2P_AGENT_GRPC_INSTANCE_ID`, and an absolute `P2P_AGENT_GRPC_SERVICE_KEY_FILE`. The client uses TLS 1.3, a mounted pairwise `DTX-Service-Key`, and the stable non-Matrix owner identifier `dirextalk-project:<server_name>`. Inline `P2P_AGENT_GRPC_SERVICE_KEY` is forbidden. Incomplete configuration or client construction fails closed and never falls back to local Chat.
- The remote slice supports ordinary prompt Chat and one deliberately narrow Cloud Dialogue scope. It forwards a UUID idempotency key, stable conversation ID, and exact conversation revision. `cloud_dialogue_mode=true` requires one owner-verified Cloud Connection and serializes only its typed connection ID; legacy `conversation_context`, `model_profile_id`, and `model_profile` remain client-compatibility inputs and are never serialized to Agent gRPC. A Chat/stream `done` response emits the optional `cloud_task` milestone only when it has the same scoped conversation ID, exactly one canonical UUID `related_task_id`, and no `related_plan_id`. The milestone is exactly `{schema:"dirextalk.cloud-agent-task/v1",task_id,conversation_id,state:"research_queued"}`: it never infers a Plan/link/lifecycle state or Cloud authority. Knowledge/Embedding, attachments, system prompts, and caller-selected tools still fail closed. A remote stream succeeds only after exactly one `done` event followed by EOF.
- The same remote connection delegates `cloud.deployments.list/get` to Agent as read-only queries. Agent derives each Deployment's Plan and Connection from its durable launch record, excludes unlinked Worker rows instead of inventing ownership, and advances the composed revision/time when either Worker or resource facts change. Message Server fully and boundedly consumes Agent pagination, validates owner/IDs/states/timestamps, and preserves the existing ProductCore response and 404 envelope. The additive `health` summary contains only status, independent health revision/times, probe counts, evidence digest, and evidence type; it never contains an endpoint, request/response data, pairing material, or secret reference. Canonical non-nil UUID managed-service records likewise come only from Agent's owner-scoped reader for `cloud.services.list/get` and the managed-service portion of `cloud.bootstrap`; Message Server never falls back to a stale local UUID row, retains only non-UUID legacy Service compatibility rows locally, and rejects a remote provider image/snapshot/volume identifier rather than projecting it. The independent Foundation establish/upgrade/teardown/remediation slice is also an owner HTTP-only compatibility façade: Message Server derives bootstrap identity inputs from an owner-scoped Role Plan or Agent Connection, forwards the exact device-bound operation fields, and keeps no Foundation or credential state. The existing owner HTTP-only `cloud.services.management.plan/approve` JSON is likewise an Agent façade: Message Server derives only the legacy Service-to-Deployment mapping and active device key, verifies Agent's byte-exact legacy signing payload, forwards the device signature, and recovers only the exact acceptance operation after response loss. Agent owns and revalidates all managed-acceptance evidence and Message Server persists no acceptance, outbox, Job or lifecycle event. The owner HTTP-only `cloud.services.managed_preparation.prepare/approve/get` actions follow the same no-store façade rule for Agent-owned managed preparation. Message Server resolves only the owner-scoped Service mapping and active signer, verifies the complete Agent scope and byte-exact deterministic CBOR, sends a signature only after a clean exact-operation NotFound, and exposes the fixed six-phase progress plus succeeded-only health/cost/stack digests. It never creates a local operation, Job, outbox, event, publish, WebSocket or MCP projection. Other Cloud mutations and query aggregates remain local until their typed façade and event projection are complete.
- The remote connection also consumes `TaskService.WatchEvents(after_seq)` with a PostgreSQL cursor keyed by Agent instance UUID and stable calling project. One transaction inserts the existing ProductCore projection, advances its aggregate revision watermark, and advances the Agent fact cursor; a failed insert cannot skip the source event, while exact replay and older revisions emit no duplicate. Reviewed projections are `cloud.plan.changed`, `cloud.task.changed`, and `cloud.step.changed`. Task/step summaries require schema v1, exact aggregate IDs, matching revision, canonical optional verified `related_plan_id`, fixed execution/outcome/stage/error enums, UTC update time, and the de-secreted owner/task/(step) fields only; unknown fields are rejected before persistence and `source_agent_instance_id` is injected. Other Agent facts advance the cursor without becoming ProductCore events until their own exact de-secreted schema exists. Agent `seq` remains the task-fact order and `p2p_events.seq` remains only projection order; both ordinary events and Agent projections allocate their final ProductCore sequence inside the same serialized database writer transaction. Changing the configured Agent instance UUID causes Flutter to rebuild only its Cloud projection, never Matrix, contact, room, or generic ProductCore state.
- A typed `CreateCloudGoal` gRPC adapter exists and validates the initial durable Task/planning facts, but the public `cloud.goals.create` action intentionally remains on its compatible `{goal, plan}` path until the Agent can atomically return a real durable Plan correlation. Message Server never fabricates a Plan ID or silently drops a selected Recipe revision during that cutover.
- Native Agent is not installed, enabled, configured, or invoked through `plugins.*`. Backend `plugins.*` actions remain for non-Agent plugins.
- Flutter still sends the legacy request-scoped model profile while the local Runner remains supported. The Message Server must not persist or return its API key, and the remote gRPC adapter discards the complete profile before crossing the Agent service boundary. Moving runtime model secrets to encrypted Agent configuration remains required before local Runner removal.
- The existing local Eino Runner includes a built-in Cloud Deployment Planner skill with `native_agent_cloud_deployment_plan`, the de-secretsed read-only `native_agent_cloud_status`, and the read-only owner-private `native_agent_cloud_recipes` recommendation tool for an explicit cloud-workload request. It requires an existing Cloud Connection before it writes a durable `researching` Goal and queued research Job. An existing Connection is established only by the owner-only CloudFormation Role Plan and signed Broker-registration flow; neither Agent nor MCP can invoke that flow. The model may recommend only de-secretsed private Recipe summaries; the client binds any selected Recipe id/revision outside model arguments. A separate Orchestrator obtains typed quotes through the Connection Stack. The Agent reads Cloud projections through narrow ports, rejects credential-shaped text, and cannot approve spend, call AWS, open ingress, operate a deployment, resume pairing, or destroy resources. Its runtime gets an isolated home and rejects direct or common wrapped AWS CLI invocation. It is not a Codex workspace Skill and it is not an external MCP tool.
- When the local Runner is selected, a request may opt into `cloud_dialogue_mode=true` for a Cloud planning conversation. This is a capability reduction, not an authorization grant: the Eino fixed allowlist contains only the research Goal tool and the two read-only status/Recipe tools that are configured; runtime shell/CLI tools, configured external MCP tools, dynamic Skill/MCP management tools, ordinary Dirextalk tools, request/config system prompts, installed Skill prompts, and conversation memory are excluded. Before a model-provider call, the server rejects credential-shaped material in `prompt`, `message`, and every request `messages[].content`/`text`; the rejection never echoes the supplied value. Client-selected top-level `cloud_connection_id` and optional `cloud_recipe_id`/`cloud_recipe_revision` are bound only in request context and omitted from the planning tool schema, so the model cannot select or replace either binding. Status-only dialogue may omit a Connection, but research planning cannot. The status tool exposes a model-minimized whitelist of Goal/Plan/Job/Deployment/Service/Alert progress and Connection state; it omits account IDs, regions, Connection identifiers, Goal prompts, artifact digests, and alert messages while adding only de-secreted `client_deep_link`, deterministic `next_step`, and the owner HTTP/device-signature control boundary. This mode may create only a connection-bound research Goal/queued Job or read that progress and never bypasses device-signed Plan approval or the typed Cloud control plane.
- Only a successful restricted planning tool result may add the optional `cloud_workload` object to the direct `agent.chat` result and final `agent.chat.stream` `done` event. Its exact schema is `dirextalk.cloud-agent-workload/v1` with only `plan_id`, `goal_id`, Plan `status`, and Plan `revision`. The server derives it from the typed Goal/Plan result, not model prose or a trace; it requires one consistent Goal/Plan relation and suppresses the field on malformed, ambiguous, non-Cloud, or failed tool output. One restricted request accepts retries of the same plan call but refuses a second distinct research goal. The object excludes the prompt, Connection ID, account/region, quote, recipe, credential/secret reference, receipt, log, endpoint, and Worker data. It remains a navigation hint rather than readiness, billing, or lifecycle evidence.

## Consumer Boundaries

- `dirextalk-connect` owns the local conversation bridge. It consumes the Matrix session and real `agent_room_id` for message sync/send and consumes the deployed `https://<server>/mcp` endpoint only through a runtime capability that supports connect-managed MCP. Host-managed runtimes keep MCP enrollment in their host runtime.
- `dirextalk-deployer` creates the Agent Matrix session, writes service-scoped `dirextalk-connect` configuration, records the canonical `/mcp` endpoint and Agent bearer credential, and generates only the runtime-specific MCP artifacts allowed by the capability registry.
- Neither consumer owns MCP business logic. They must not recreate a local MCP CLI, daemon, proxy, stdio bridge, listening port, fixed `mcp.*` product action path, or alternate endpoint.
- Flutter reads Online Agent availability from Matrix state in `agent_room_id` and uses backend-owned `agent.*` actions and native stream frames for Native Agent. It does not call fixed `mcp.*` product actions.

## vNext Legacy Matrix Gateway Foundation (Release Gate M)

- The internal Gateway adapter accepts only owner-authored `io.dirextalk.agent.invoke.v1` timeline events from the configured real `agent_room_id`. Its consumer uses an independent JetStream durable, so an Agent Control outage cannot block normal ProductCore projections.
- Invoke content is capped at 16 KiB and strictly contains `request_id`, `installation_id`, optional `preferred_connector_id`, `dispatch_mode`, `grant_version`, `input_event_id`, `required_capabilities`, and `idempotency_key`. UUIDs are canonical UUIDv7; capabilities are bounded, lowercase, unique, and sorted. Unknown/duplicate fields, trailing JSON, unsafe grant versions, the wrong room, and non-owner senders are ignored without creating a Run.
- PostgreSQL migration v38 stores one reservation per `(matrix_room_id, request_id)`, with unique source event and tenant/room/idempotency digest constraints. It stores the local Matrix input reference and normalized routing facts, but never the prompt body or raw idempotency key. Crash replay returns the first generated opaque request event and request digest; accepted/rejected terminal facts are source-digest fenced and immutable.
- Agent Control is called through the frozen `dirextalk.agent_gateway.v1.AgentRunIngress/CreateAgentRun` contract. The client requires TLS 1.3, HTTP/2, explicit server roots, a clientAuth-only certificate with exactly `spiffe://dirextalk.internal/v1/tenants/<tenant>/services/legacy-matrix-gateway`, 64 KiB message limits, and a deadline no longer than 20 seconds. Request and idempotency digests use the frozen LP/COMMIT transcript.
- The production monolith does not expose a startup switch for this adapter yet. Activation remains deliberately unavailable until deployment can prove the old Connect room consumer is stopped and fenced; otherwise one Matrix input could execute through both paths.
- This foundation durably creates or replays an Agent Run. Exclusive-consumer cutover, Run completion/evidence ingress, `io.dirextalk.agent.result.v1` / `io.dirextalk.agent.error.v1` projection, and restricted plain-text fallback remain later Release Gate M work; the server must not fabricate completion or evidence from an admission receipt.
